# Distribution Spec

Status: deferred by design. The newsroom core records and compiles distribution-ready editorial output, but this repo does not execute outbound distribution yet.

## Purpose

The distribution layer will eventually consume approved newsroom records and package them for off-site channels without introducing a remote CMS or a separate database.

Until implementation exists:

- newsroom content stays file-based under `newsroom/`
- compiled site output is generated by `npm run newsroom:build`
- distribution execution must not post, send, enqueue, or retry anything outward
- any future queue artifacts should be written under `newsroom/generated/distribution-queue/`

## Input Contract

Distribution may only ingest records that satisfy all of the following:

- article or dispatch record exists under `newsroom/content/`
- record status is `published`
- required approval records exist in `newsroom/approvals/`
- record is present in `newsroom/generated/site-content.json`
- `distributionStatus` on the source record remains `deferred` until a future executor explicitly claims it

Primary source record types:

- `article`
- `dispatch`

Required source fields for articles:

- `id`
- `slug`
- `headline`
- `dek`
- `excerpt`
- `section`
- `author`
- `publishedAt` or `publishAt`
- `approvalIds`
- `distributionStatus`

Required source fields for dispatch issues:

- `id`
- `slug`
- `issueTitle`
- `issueNumber`
- `editorsNote`
- `publishDate`
- `leadStorySlug`
- `items`
- `approvalIds`
- `distributionStatus`

## Queue Model

Future queue items should be file-based JSON objects written to `newsroom/generated/distribution-queue/`.

Recommended filename pattern:

- `YYYY-MM-DDTHH-mm-ssZ-<channel>-<slug>.json`

Recommended queue record shape:

```json
{
  "id": "dist-x-rest-defense-is-learning-to-breathe",
  "createdAt": "2026-04-06T12:00:00.000Z",
  "channel": "x",
  "sourceKind": "article",
  "sourceId": "article-rest-defense-breathe",
  "sourceSlug": "rest-defense-is-learning-to-breathe",
  "status": "queued",
  "attemptCount": 0,
  "payload": {
    "headline": "Barca's Rest Defense Is Learning to Breathe",
    "url": "https://twotalbarca.com/article/rest-defense-is-learning-to-breathe",
    "summary": "A tactical read on the calmer shape Barca is finally leaving behind the ball."
  },
  "review": {
    "approvedBy": "editor-in-chief",
    "approvedAt": "2026-04-06T12:00:00.000Z"
  }
}
```

Permitted queue statuses:

- `queued`
- `processing`
- `sent`
- `failed`
- `canceled`

## Channel Model

Initial target channels to support later:

- `x`
- `telegram`
- `linkedin`
- `instagram`
- `newsletter`
- `rss-amplification`

Each channel adapter should declare:

- character or asset constraints
- whether link previews are supported
- whether media is required or optional
- whether human review is mandatory
- rate-limit expectations

## Review And Approval Gates

No future executor should send outbound content unless:

- the source newsroom record is `published`
- the copy gate approval exists
- the editor approval exists
- channel packaging is derived from the approved thesis rather than new claims
- any channel-specific copy has either:
  - explicit editor approval captured in the queue item, or
  - a deterministic rule showing it was derived from approved source text without adding claims

Additional gate for breaking-news or matchday packaging:

- facts that are time-sensitive must be re-verified before send

## Retry, Logging, And Audit

When implementation arrives, it should log locally and append-only.

Minimum expectations:

- every send attempt records `attemptedAt`
- every failure records a machine-readable error code and human-readable message
- retries use bounded backoff
- permanent failures stop after a configured max-attempt threshold
- successful sends record returned provider ids when available
- logs remain file-based and repo-local or local-runtime-local unless a future design explicitly changes that

Recommended local log locations:

- `newsroom/generated/distribution-queue/`
- `newsroom/generated/distribution-logs/`

## Analytics Expectations

Analytics are also deferred, but the contract should plan for:

- sent count by channel
- failure count by channel
- retry count by channel
- source record to channel mapping
- optional downstream engagement fields captured later without blocking send

Any future analytics collector must not be required for the newsroom core to publish to the site.

## Non-Goals

Not implemented in this phase:

- outbound API calls
- credential management for distribution platforms
- channel-specific media generation
- scheduled send workers
- analytics dashboards
- reader list management

## Current Handoff

Today the newsroom stops at:

1. validate newsroom state
2. publish approved records into `newsroom/generated/site-content.json`
3. reference this document for any deferred distribution planning

That boundary is intentional. The autonomous newsroom core is complete without outbound distribution execution.

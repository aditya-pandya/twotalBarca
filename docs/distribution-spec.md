# totalBarca Distribution Spec

Last updated: 2026-05-02
Status: deferred by design. The repo prepares weekly issue output locally but does not execute outbound distribution.

## 1. Purpose

If distribution is ever implemented later, it should package the weekly dispatch cleanly without turning totalBarca into a daily churn machine.

Until implementation exists:
- newsroom content stays file-based under `newsroom/`
- compiled site output is generated by `npm run newsroom:build`
- no code in this repo should post, send, enqueue, or retry anything outward
- any future queue artifacts should live under `newsroom/generated/distribution-queue/`

## 2. Active future input contract

Primary source record:
- `dispatch`

Required source conditions:
- record exists under `newsroom/content/dispatch/`
- record status is `published`
- required approval records exist in `newsroom/approvals/`
- record is present in `newsroom/generated/site-content.json`
- `distributionStatus` remains `deferred` until a future executor explicitly claims it

Active required fields for a future dispatch send:
- `id`
- `slug`
- `issueTitle`
- `issueNumber` or stable issue label
- `publishDate`
- `topics` (must be length 5)
- `lastMatch`
- `nextMatch`
- `approvalIds`
- `distributionStatus`

## 3. Optional secondary packaging later

A future system may derive:
- one dispatch summary post
- one archive/index update
- one lightweight reminder or teaser tied to the weekly issue

Not active even for future planning by default:
- live match alerts
- daily Brief sends
- broad article amplification
- channel spam derived from each of the five topics as separate feed items

## 4. Queue model

Future queue items should remain file-based JSON objects written to `newsroom/generated/distribution-queue/`.

Recommended filename pattern:
- `YYYY-MM-DDTHH-mm-ssZ-<channel>-<dispatch-slug>.json`

Recommended queue record shape:

```json
{
  "id": "dist-newsletter-weekly-dispatch-18",
  "createdAt": "2026-05-08T12:00:00.000Z",
  "channel": "newsletter",
  "sourceKind": "dispatch",
  "sourceSlug": "weekly-dispatch-issue-18",
  "status": "queued",
  "attemptCount": 0,
  "payload": {
    "title": "Weekly Dispatch No. 18",
    "summary": "Five sharp reads on Barça plus minimal match context.",
    "url": "https://totalbarca.com/dispatch/weekly-dispatch-issue-18"
  },
  "review": {
    "approvedBy": "editor-in-chief",
    "approvedAt": "2026-05-08T12:00:00.000Z"
  }
}
```

## 5. Review and approval gates

No future executor should send anything unless:
- the source dispatch record is `published`
- copy approval exists
- editor approval exists
- channel packaging does not add new claims beyond the approved weekly issue
- time-sensitive match facts are rechecked before send if they could have moved

## 6. Non-goals

Not implemented in this phase:
- outbound API calls
- credential management
- scheduled send workers
- daily content queues
- analytics dashboards
- reader-list management

## 7. Current handoff boundary

Today the repo stops at:
1. validate newsroom state
2. publish approved weekly issue data into `newsroom/generated/site-content.json`
3. reference this document for any deferred distribution planning

That boundary is intentional.
